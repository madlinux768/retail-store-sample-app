DevOps Agent Observability Guide
Step-by-step log investigation paths for each alarm scenario the agent may encounter
🔴

Connectivity Failure

Alarm: cross-vpc-connectivity-down
  1. Check /aws/lambda/networking-demo-connectivity-check for the most recent invocation. The error_message field narrows the root cause immediately: timed out (routing/TGW) vs Connection refused (SG/host) vs Name or service not known (DNS).
  2. Check /aws/tgw/networking-demo for the relevant attachment IDs. Zero packets for traffic toward 10.1.0.0/16 confirms a blackhole route or missing TGW route table entry.
  3. Check /aws/vpc/networking-demo-partner with filter action=REJECT and dstport=80. REJECT records pointing to the partner EC2 IP confirm a security group block.
  4. Correlate timestamps across all three sources to confirm a single root cause before proposing a fix.
🟠

DNS Resolution Failure

Alarm: dns-connectivity-down
  1. Check /aws/route53resolver/networking-demo for queries to api.partner.internal. An rcode=NXDOMAIN confirms the PHZ record is missing or the zone is not associated with the VPC.
  2. An rcode=SERVFAIL suggests the PHZ association itself was removed. Verify the PHZ Z05579592LTL36F75RLS5 is still associated with the app VPC via the Route 53 console or AWS CLI.
  3. Confirm Lambda logs show Name or service not known — this corroborates DNS failure rather than a downstream TCP issue.
  4. If PHZ association is intact, check that the NLB DNS alias record exists in the zone and the NLB itself is healthy.
🟡

Latency Spike

Alarm: cross-vpc-latency-high
  1. Check /aws/lambda/networking-demo-connectivity-check for the latency_ms and dns_resolution_ms fields. A high dns_resolution_ms with normal TCP latency isolates the issue to DNS (resolver overload or TTL expiry storm).
  2. Check /aws/tgw/networking-demo for abnormal packet counts or retransmission patterns on the attachment toward 10.1.0.0/16 — packet loss at the TGW level causes TCP retransmits and visible latency.
  3. Check S3 NLB access logs (networking-demo-nlb-logs-173471018689) for elevated target_processing_time — if the NLB shows normal connection time but high target time, the partner EC2 is CPU-bound or overwhelmed.
  4. Check VPC flow logs for flow-direction=ingress on the NLB ENIs toward 10.1.10.163 to confirm traffic is reaching the partner subnet.
🟣

NLB Unhealthy Targets

Alarm: nlb-unhealthy-targets
  1. Check S3 NLB access logs (networking-demo-nlb-logs-173471018689) for connection_log entries with action=fail on health check probes to 10.1.10.163:80.
  2. Check /aws/vpc/networking-demo-partner for return traffic from 10.1.10.163 back to the NLB ENI source IPs. No return packets (asymmetric routing) means health checks reach the EC2 but responses are dropped.
  3. Check the partner security group sg-0c4f12bf1bc479afe: VPC flow logs with action=REJECT on port 80 from NLB ENI source IPs confirm a security group block of health checks.
  4. If SG is open, check EC2 instance status (i-0ae35fba922471a0e) — a stopped instance will show TCP connection resets in the NLB logs.
